# Безопасный сценарий авторизации для баланса бонусной карты

Навык не должен запрашивать номер карты, пароль, код из SMS или другие секреты голосом. Баланс бонусной карты нужно показывать только после штатной привязки аккаунта через OAuth/account linking и проверки токена на backend кофейни.

## Когда запускать `start_account_linking`

Запускайте `start_account_linking` только тогда, когда пользователь запросил персональные данные, а у навыка еще нет действующего токена. Например:

- пользователь говорит: "Покажи баланс", "Сколько бонусов на карте", "Какие у меня баллы";
- в запросе нет `access_token`;
- устройство и клиент поддерживают account linking.

Не стоит запускать привязку сразу при первом открытии навыка, если пользователь еще не попросил действие, требующее аккаунта. Для общих вопросов вроде меню, адресов кофеен или акций авторизация не нужна.

Типовой поток:

1. Распознать намерение `show_bonus_balance`.
2. Проверить, есть ли OAuth-токен в запросе.
3. Если токена нет, проверить поддержку account linking.
4. Если поддержка есть, сохранить исходное намерение и вернуть ответ со `start_account_linking`.
5. После успешной привязки получить новый запрос уже с токеном, проверить токен на backend и показать баланс.

## Что проверить в `meta.interfaces`

Перед возвратом `start_account_linking` проверьте, что в запросе есть поддержка интерфейса привязки аккаунта:

```json
{
  "meta": {
    "interfaces": {
      "account_linking": {}
    }
  }
}
```

Проверка должна быть именно по наличию соответствующего ключа в `meta.interfaces`. Если `meta.interfaces.account_linking` отсутствует, клиент пользователя не сможет обработать директиву привязки, и навык должен пойти по fallback-сценарию.

## Где искать `access_token`

После успешной привязки OAuth-токен приходит в объекте пользователя в сессии:

```json
{
  "session": {
    "user": {
      "user_id": "...",
      "access_token": "..."
    }
  }
}
```

Backend навыка должен читать токен из `session.user.access_token`. Нельзя считать пользователя авторизованным только по `user_id`: это стабильный идентификатор пользователя Алисы для навыка, но не подтверждение доступа к аккаунту кофейни.

Перед показом баланса нужно:

- проверить токен на backend кофейни или через introspection/userinfo endpoint;
- убедиться, что у токена есть нужные scope, например доступ к бонусной карте;
- обработать истечение срока действия, отзыв токена и ошибки авторизации как отсутствие действующей привязки;
- не логировать токен целиком и не передавать его в тексте ответа.

## Если account linking недоступен

Если в `meta.interfaces` нет `account_linking`, не пытайтесь собирать учетные данные в диалоге. Безопасный ответ:

> Чтобы показать баланс, нужно войти в аккаунт кофейни. На этом устройстве привязка аккаунта недоступна. Откройте навык в приложении Яндекса или проверьте баланс в приложении кофейни.

Можно предложить неперсональные альтернативы: рассказать, как работает бонусная программа, где найти карту в приложении, или назвать ссылку/кнопку на сайт, если формат клиента это поддерживает. Но сам баланс в этой ветке показывать нельзя.

## Как не потерять исходный запрос после авторизации

Перед отправкой `start_account_linking` сохраните исходное действие в серверном состоянии:

```json
{
  "user_id": "alice-user-id",
  "pending_action": "show_bonus_balance",
  "slots": {},
  "created_at": "2026-06-21T12:00:00Z"
}
```

Хранить лучше минимальные данные: намерение, необходимые параметры и время создания. Не кладите в состояние токены, пароли, номера карт или лишние персональные данные.

После того как пользователь вернется из авторизации и навык получит запрос с `session.user.access_token`:

1. Найдите `pending_action` по `session.user.user_id` или другому безопасному ключу состояния.
2. Проверьте, что запись свежая, например не старше 10-15 минут.
3. Проверьте OAuth-токен.
4. Выполните исходное действие: запросите баланс у backend кофейни и ответьте пользователю.
5. Удалите `pending_action`, чтобы действие не повторилось случайно.

Если сохраненное действие устарело или не найдено, не угадывайте намерение. Скажите: "Готово, аккаунт привязан. Что хотите узнать?" Если действие найдено, но прошло слишком много времени, безопаснее уточнить: "Аккаунт привязан. Показать баланс бонусной карты?"

## Итоговый безопасный сценарий

Для запроса "Покажи баланс бонусной карты" backend навыка должен сначала искать `session.user.access_token`. Если токен есть, он проверяет его у кофейни и возвращает баланс. Если токена нет, backend проверяет `meta.interfaces.account_linking`; при наличии интерфейса сохраняет исходное намерение и возвращает `start_account_linking`. Если интерфейс недоступен, навык объясняет, что привязка на этом устройстве невозможна, и предлагает безопасный альтернативный способ входа. После авторизации навык восстанавливает сохраненный запрос, проверяет токен и только затем показывает баланс.
