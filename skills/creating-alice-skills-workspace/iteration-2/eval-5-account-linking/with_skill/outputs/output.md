# Безопасный сценарий авторизации для баланса бонусной карты

Баланс бонусной карты — персональные данные аккаунта кофейни, поэтому навык не должен называть баланс до проверки авторизации. Авторизацию лучше запускать не сразу при открытии навыка, а в момент, когда пользователь просит приватное действие: «покажи баланс», «сколько бонусов», «моя карта». До этого навык может без авторизации рассказывать про меню, адреса, правила программы лояльности и способы получить карту.

## Поток обработки запроса

1. Распознать намерение пользователя, например `check_bonus_balance`.
2. Сохранить исходное намерение и параметры в состояние до запуска авторизации:

```json
{
  "session_state": {
    "pending_action": {
      "intent": "check_bonus_balance",
      "original_command": "покажи баланс бонусной карты"
    }
  }
}
```

3. Проверить, есть ли уже токен. Основной источник — заголовок webhook-запроса:

```http
Authorization: Bearer <access_token>
```

Если платформа или облачная функция не передает заголовки, искать токен в теле запроса:

```json
{
  "session": {
    "user": {
      "access_token": "<access_token>"
    }
  }
}
```

4. Если токен есть, проверить его на ресурсном сервере кофейни или вызвать API баланса с этим токеном. Если токен валиден, вернуть баланс. Если токен отсутствует, просрочен или отклонен ресурсным сервером, переходить к авторизации.

## Когда запускать `start_account_linking`

Запускать `start_account_linking` нужно только когда одновременно выполнены условия:

- пользователь запросил данные, требующие аккаунта кофейни;
- валидного `access_token` нет;
- текущая поверхность поддерживает связку аккаунтов.

Перед отправкой директивы обязательно проверить:

```js
const canLink = Boolean(request.meta?.interfaces?.account_linking);
```

Если `meta.interfaces.account_linking` есть, навык отвечает с директивой:

```json
{
  "response": {
    "text": "Чтобы показать баланс бонусной карты, нужно войти в аккаунт кофейни.",
    "end_session": false,
    "directives": {
      "start_account_linking": {}
    }
  },
  "session_state": {
    "pending_action": {
      "intent": "check_bonus_balance",
      "original_command": "покажи баланс бонусной карты"
    }
  },
  "version": "1.0"
}
```

Токен не нужно запрашивать у пользователя голосом и нельзя просить назвать пароль или код. Пользователь вводит учетные данные только на странице авторизации кофейни в OAuth-потоке.

## Если `account_linking` недоступен

Если `meta.interfaces.account_linking` отсутствует, не отправлять `start_account_linking`: такая команда может не сработать на поверхности пользователя. Вместо этого нужно дать безопасный fallback:

- объяснить, что на этом устройстве вход в аккаунт недоступен;
- не раскрывать баланс;
- предложить открыть навык на поддерживаемой поверхности или привязать аккаунт в приложении/каталоге навыка;
- оставить сценарий без авторизации доступным, например рассказать правила программы лояльности.

Пример ответа:

```json
{
  "response": {
    "text": "Я могу показать баланс только после входа в аккаунт кофейни, но на этом устройстве авторизация недоступна. Откройте навык в приложении Яндекс или Яндекс Браузере и привяжите аккаунт, затем снова попросите баланс.",
    "end_session": false
  },
  "version": "1.0"
}
```

## Как не потерять исходный запрос после авторизации

Перед `start_account_linking` сохранять не только текст команды, а нормализованное намерение: `pending_action.intent`, нужные параметры и `original_command`. Для краткого сценария достаточно `session_state`, но если пользователь может завершить авторизацию и вернуться в новый сеанс, лучше продублировать безопасный маркер ожидания в пользовательское состояние через `user_state_update`. Не хранить там токен, пароль, номер карты целиком или другие секреты.

После успешной связки аккаунтов следующие запросы к webhook будут приходить уже с `Authorization: Bearer <access_token>` или `session.user.access_token`. На новом обращении навык должен:

1. снова прочитать токен и проверить его;
2. посмотреть сохраненный `pending_action`;
3. если там `check_bonus_balance`, выполнить исходный запрос автоматически или спросить короткое подтверждение: «Аккаунт подключен. Показать баланс бонусной карты?»;
4. после успешного показа баланса очистить `pending_action`, чтобы старый запрос не выполнился повторно.

Так пользователь не должен заново формулировать «покажи баланс», а навык при этом не раскрывает персональные данные без валидного OAuth-токена.
